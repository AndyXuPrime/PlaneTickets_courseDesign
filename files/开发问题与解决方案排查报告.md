# 机票预订系统开发问题与解决方案排查报告

> 本报告基于 `files` 目录下的阶段方案、阶段总结、系统架构文档和设计书整理，重点保留开发过程中真实记录过的现象、原因和处理方式。部分内容是对多个阶段文档中重复问题的归并，不作为当前代码的最终实现说明；最终实现仍以项目 README、架构文档和实际代码为准。

## 1. 项目排障背景

本项目从单体 Java Web 逐步重构为微服务机票预订系统，后端主要由 `gateway-service`、`auth-service`、`flight-service`、`order-service`、`admin-service` 和 `common-module` 组成，配合 Nacos、OpenFeign、Redis、MinIO、MySQL 等组件；前端采用 Vue 2、Element UI、Axios 与 ECharts。

开发中遇到的问题主要集中在以下几类：

| 类型 | 高频表现 |
| --- | --- |
| Maven 与模块依赖 | Common 模块改动不生效、找不到类、插件缺失、旧 Jar 被引用 |
| Nacos 与配置中心 | 9848 端口连接失败、配置拉取失败、JWT Key 缺失、数据库名写错 |
| Gateway 技术栈冲突 | WebFlux 与 Servlet 过滤器/异常处理器冲突，导致 500 或过滤器不生效 |
| 鉴权与跨服务调用 | Feign 调用被 Spring Security 拦截，出现 401/403 |
| Feign 接口契约 | 返回类型不一致、数据库无数据，导致 DecodeException 或 404 |
| 数据库与 JPA | 保留字、懒加载、约束校验、加密字段长度不足 |
| Redis 缓存 | 缓存没有进 Redis、LocalDate 序列化导致 500 |
| 前端工程化 | ECharts 依赖找不到、ESLint 报错、样式穿透、响应式布局挤压 |

## 2. 快速排查索引

本索引按“实际开发/排障链路”排序：先看项目能不能构建和启动，再看配置中心，再看请求是否能过 Gateway，随后排查鉴权与服务间调用，最后处理数据层、缓存和前端问题。

### 2.1 构建与模块依赖

| 日志/现象 | 优先怀疑 | 当时解决方式 |
| --- | --- | --- |
| `No plugin found for prefix 'spring-boot'` | 子模块缺少 Spring Boot Maven 插件 | 在对应子模块 `pom.xml` 补全 `<build><plugins>...</plugins></build>` |
| `package com.bighomework.common... does not exist` / `Symbol not found` | `common-module` 未安装到本地 Maven 仓库，或包名迁移不完整 | 根目录执行 `mvn clean install -DskipTests`，并修正 `package` 与 `import` |
| `NoSuchMethodError` / DTO 字段为 `null` | 子服务运行的是旧版 Common Jar | 修改 Common 后强制执行 `mvn clean install -DskipTests`，再重启相关服务 |

### 2.2 配置中心、数据库连接与服务注册

| 日志/现象 | 优先怀疑 | 当时解决方式 |
| --- | --- | --- |
| `Connection refused ... 9848` | Nacos 2.x gRPC 端口未通 | 确认 Nacos standalone 启动，检查 9848 端口和防火墙 |
| `Failed to configure a DataSource` 且 Nacos 连接失败 | 配置未从 Nacos 拉取，导致数据库 URL 缺失 | 先修复 Nacos 连接，再检查服务配置 |
| `Unknown database 'plane_ticket_db'` | JDBC URL 中数据库名与本地 Schema 不一致 | 将 Nacos 中数据库名改为实际的 `planetickets` |
| `Could not resolve placeholder 'app.jwt.secret'` | 引入了 `JwtTokenProvider`，但当前服务缺 JWT 配置 | 在对应服务的 Nacos YAML 中补齐 `app.jwt.secret` 和过期时间 |
| Gateway 转发 500，注册 IP 是 `192.168.x.x` | Nacos 注册到了虚拟网卡 IP | 在服务 Nacos 配置中指定 `spring.cloud.nacos.discovery.ip: 127.0.0.1` |

### 2.3 Gateway 启动、路由与跨域

| 日志/现象 | 优先怀疑 | 当时解决方式 |
| --- | --- | --- |
| Gateway 启动失败或过滤器不生效 | 使用了 Servlet 过滤器，但 Gateway 基于 WebFlux | 删除 Servlet Filter，改写为 `GlobalFilter` |
| Gateway 报 500 且无日志 | WebFlux Gateway 扫描了 Servlet 版 `GlobalExceptionHandler` | 调整 `GatewayApplication` 的扫描范围，避免扫描 `common.exception` |
| Gateway 引入 Common 后尝试连数据库 | Common 间接引入 JPA/DataSource 自动配置 | Gateway 启动类排除 `DataSourceAutoConfiguration`，并控制依赖引入 |
| 前端跨域报 `blocked by CORS policy` | 预检请求被网关拦截或 CORS 配置与凭证冲突 | Nacos 中配置 `allowedOriginPatterns: "*"`，并开启预检映射相关配置 |
| `allowedOrigins cannot contain * when allowCredentials is true` | `allowCredentials=true` 时不能使用 `allowedOrigins: "*"` | 改为 `allowedOriginPatterns: "*"` |

### 2.4 Auth 鉴权与 Feign 服务联调

| 日志/现象 | 优先怀疑 | 当时解决方式 |
| --- | --- | --- |
| Auth 登录 500 | `auth-service.yaml` 缺数据库、Redis 或 JWT 配置 | 补齐 Auth 服务 Nacos 配置，尤其是 JWT 配置 |
| Auth 登录 401 | 数据库手动插入了明文密码 | 通过注册流程生成 BCrypt 密码，登录时用 `matches` 校验 |
| Feign 调用 401/403 | 下游服务默认 Security 拦截了内部调用 | 添加 `SecurityConfig` 放行内部接口，或仅保留 Gateway 统一鉴权 |
| Feign 调用 404 | 数据库没有对应航班等测试数据 | 插入真实可查询的数据，再联调接口 |
| Feign `DecodeException` | Controller 返回 `List`，Feign Client 定义为单对象或 `Object` | 对齐 Controller 与 Feign Client 的返回类型，必要时新增单对象接口 |

### 2.5 数据库、JPA、缓存与业务一致性

| 日志/现象 | 优先怀疑 | 当时解决方式 |
| --- | --- | --- |
| Redis `keys *` 为空 | Spring Cache 没有指定 Redis，退化成本地缓存 | 配置 `spring.cache.type: redis`，并开启 `@EnableCaching` |
| Redis 开启后接口 500 | `LocalDate` 等 Java 8 时间类型不能被默认序列化 | 自定义 `RedisConfig`，注册 `JavaTimeModule` |
| 航班列表 `StackOverflowError` | JPA 双向关联被 Jackson 无限递归序列化 | 在关联字段加 `@JsonIgnoreProperties` 或 `@JsonIgnore` |
| 保存订单 SQL 语法错误 | `tickets.class` 使用了数据库/Java 保留字 | 实体映射中对 `class` 字段做转义，或后续改用非保留字字段名 |
| 用户查航班 500，但管理员正常 | VO 转换时访问懒加载对象，Session 已关闭 | Service 层加 `@Transactional(readOnly = true)`，Repository 使用 `JOIN FETCH` |
| 注册用户时崩溃 | AES 加密后身份证密文超过 `varchar(18)` | 将数据库和实体 `idCard` 长度扩容到 `255` |
| 退票时报 `chk_valid_payment_time` 约束错误 | 状态改为已取消，但 `payment_time` 未清空 | `refundTicket` 中显式执行 `ticket.setPaymentTime(null)` |
| 乘机人姓名都变成订票人 | 前后端 ID 类型不一致，且虚拟 ID 后端无法匹配 | 前端解析乘机人姓名，直接传 `passengerNames` 给后端 |

### 2.6 前端依赖、编译与界面适配

| 日志/现象 | 优先怀疑 | 当时解决方式 |
| --- | --- | --- |
| `Module not found: Can't resolve 'echarts'` | 在错误目录安装依赖，或前端子项目缺依赖 | 进入 `frontend` 目录执行 `npm install echarts --save` 并重启 Dev Server |
| ESLint `x-invalid-end-tag` / `no-empty` | 模板标签嵌套错误，或空 `catch` 块 | 扁平化模板结构，给 `catch` 补充日志 |

## 3. 后端构建与公共模块问题

### 3.1 Spring Boot Maven 插件缺失

**现象**：`gateway-service` 启动时报 `No plugin found for prefix 'spring-boot'`。

**原因**：子模块 `pom.xml` 没有配置 Spring Boot Maven 插件，导致不能直接运行 `spring-boot:run`。

**解决**：在对应子模块 `pom.xml` 中补全 `<build><plugins>...</plugins></build>`。

**排查建议**：

1. 先确认命令是在正确模块下执行。
2. 检查子模块是否继承父工程版本管理。
3. 确认子模块 `pom.xml` 是否包含 Spring Boot Maven Plugin。

### 3.2 Common 模块修改后不生效

**现象**：其他服务编译时报 `package ... does not exist`、`Symbol not found`，运行时报 `NoSuchMethodError`，或 DTO 新字段始终为 `null`。

**原因**：

1. `common-module` 作为 Jar 被其他服务依赖，修改源码后如果没有安装到本地 Maven 仓库，其他服务仍会引用旧版 Jar。
2. 文件移动后，Java 文件内的 `package` 声明和 `import` 语句仍指向旧路径。
3. 父工程与子模块 `pom.xml` 职责不清，父工程误放了子模块依赖。

**解决**：

1. 明确父工程只管理版本和模块，子模块自己声明业务依赖。
2. 全局修正 `package` 与 `import`。
3. 每次修改 Common 后，在根目录执行：

```bash
mvn clean install -DskipTests
```

4. 重启依赖 Common 的服务。

**经验沉淀**：形成“改 Common -> 根目录 install -> 重启子服务”的固定流程。这条规则在后续 DTO、Feign 接口、JWT 工具类调整中反复验证有效。

## 4. Nacos 与配置中心问题

### 4.1 Nacos 9848 端口连接失败

**现象**：应用启动日志出现 `Connection refused ... 9848`、`Client not connected`，随后可能连带出现 `Failed to configure a DataSource`。

**原因**：Nacos 2.x 客户端除 HTTP 端口外，还需要连接 gRPC 端口 9848。如果 Nacos 未启动、未以 standalone 模式运行、端口被防火墙拦截，应用就无法拉取配置。配置拉取失败后，数据库 URL 等配置缺失，Spring Boot 会继续抛出 DataSource 自动配置错误。

**解决**：

1. 确认 Nacos 正常启动，开发环境使用 standalone 模式。
2. 检查 9848 端口是否可用。
3. 本地 `bootstrap.yml` 只保留能找到 Nacos 的最小配置。
4. Nacos 恢复后，再观察具体服务配置是否完整。

### 4.2 数据库 Schema 名称不一致

**现象**：启动或访问数据库时报 `Unknown database 'plane_ticket_db'`。

**原因**：Nacos 配置中的 JDBC URL 指向 `plane_ticket_db`，但本地实际数据库名为 `planetickets`。

**解决**：修改对应服务的 Nacos YAML，将 JDBC URL 中的数据库名改为 `planetickets`。

**排查建议**：这类问题不要先改代码，优先核对 Nacos 配置、MySQL Schema 名称和本地连接串。

### 4.3 JWT 配置缺失

**现象**：Gateway、Flight 或 Auth 启动时报 `Could not resolve placeholder 'app.jwt.secret'`，Auth 登录可能直接 500。

**原因**：`common-module` 中的 `JwtTokenProvider` 依赖 `app.jwt.secret` 等配置。哪个服务引用了这个工具类，哪个服务就必须在自己的配置中提供对应配置。

**解决**：

1. 在 `gateway-service.yaml`、`flight-service.yaml`、`auth-service.yaml` 等引用 JWT 工具类的配置中补齐：
   - `app.jwt.secret`
   - `app.jwt.expiration-ms`
2. Auth 服务还要确认数据库和 Redis 配置齐全。

**经验沉淀**：公共模块中的 `@Value` 配置项会“跟着依赖走”。只要某服务依赖了使用该配置项的 Bean，就要为该服务补配置。

### 4.4 Nacos 注册 IP 漂移

**现象**：Gateway 转发 500，日志中服务注册 IP 是 `192.168.x.x` 虚拟网卡地址。

**原因**：本机存在虚拟网卡，服务注册到 Nacos 时选择了不可被 Gateway 正确访问的 IP。

**解决**：在所有本地服务的 Nacos YAML 中显式指定：

```yaml
spring:
  cloud:
    nacos:
      discovery:
        ip: 127.0.0.1
```

**排查建议**：如果服务已注册但 Gateway 调不到，除了看路由，也要去 Nacos 控制台检查实例 IP。

## 5. Gateway 与 WebFlux 技术栈问题

### 5.1 Servlet Filter 与 WebFlux Gateway 冲突

**现象**：Gateway 启动失败，或 JWT 过滤器不生效。

**原因**：原有 `JwtAuthenticationFilter` 基于 Servlet/Tomcat，而 Spring Cloud Gateway 基于 WebFlux/Netty，底层模型不兼容。

**解决**：

1. 删除 Gateway 中旧的 Servlet 过滤器。
2. 编写实现 `GlobalFilter` 的 `AuthGlobalFilter`。
3. 在过滤器中完成 JWT 校验，解析 `UserId`、`Role`、`AirlineCode`，再写入下游 Header，例如 `X-User-Name`、`X-User-Role`、`X-Airline-Code`。

### 5.2 Gateway 500 但没有日志

**现象**：Gateway 报 500，但没有有效异常日志。

**原因**：Gateway 是 WebFlux 应用，却扫描到了 `common-module` 中基于 Servlet MVC 的 `GlobalExceptionHandler`，两套异常处理模型冲突。

**解决**：调整 `GatewayApplication` 的 `scanBasePackages`，避免扫描 `common.exception` 等 Servlet 相关异常处理器。

**排查建议**：Gateway 项目中如果出现无日志 500，优先检查扫描范围、Starter 依赖和是否混入 Servlet 组件。

### 5.3 Gateway 引入 Common 后触发数据库自动配置

**现象**：Gateway 引入 `common-module` 后，启动时尝试配置数据库，随后因没有数据源配置报错。

**原因**：Common 或其依赖链间接引入了 JPA/DataSource 相关自动配置。Gateway 作为流量入口，本身不需要直连数据库。

**解决**：

1. 在 Gateway 启动类上排除 `DataSourceAutoConfiguration`。
2. 在 Gateway 的依赖中控制或排除不需要的 Web/JPA 依赖。

**经验沉淀**：Gateway 要尽量轻，避免把业务层、持久层组件带入网关。

## 6. 跨域与前端请求问题

### 6.1 CORS 预检请求被拦截

**现象**：前端 8085 端口访问 Gateway 8080 时，经常出现 `blocked by CORS policy`，即使配置了跨域仍不稳定。

**原因**：浏览器会先发送 `OPTIONS` 预检请求。如果 Gateway 没有正确处理预检请求，或者路由匹配不到预检请求，正式请求还没发出就会失败。

**解决**：

1. 在 Nacos 的 Gateway 配置中开启预检请求处理相关开关，例如 `add-to-simple-url-handler-mapping: true`。
2. 使用 `allowedOriginPatterns: "*"` 替代 `allowedOrigins: "*"`。

### 6.2 `allowedOrigins` 与凭证配置冲突

**现象**：日志出现 `IllegalArgumentException: ... allowedOrigins cannot contain * when allowCredentials is true`。

**原因**：当 `allowCredentials` 为 `true` 时，Spring 不允许 `allowedOrigins` 使用通配符 `*`。

**解决**：将配置改为：

```yaml
allowedOriginPatterns: "*"
```

**排查建议**：跨域问题优先看浏览器控制台、Network 中的 `OPTIONS` 请求、Gateway Nacos 配置三处。

## 7. Auth 与安全鉴权问题

### 7.1 Auth 登录 500

**现象**：登录接口返回 500。

**原因**：阶段记录中判断主要是 `auth-service` 引用了 Common 模块，但 Nacos 配置不完整，尤其缺少 JWT 配置。

**解决**：

1. 登录 Nacos。
2. 新建或编辑 `auth-service.yaml`。
3. 确认至少包含数据库配置、Redis 配置和 JWT 配置。

### 7.2 Auth 登录 401

**现象**：接口可访问，但登录失败并返回 401。

**原因**：数据库中手动插入了明文密码，而登录逻辑使用 BCrypt 校验。明文与 BCrypt 哈希不能匹配。

**解决**：

1. 注册新用户，让系统通过 `BCryptPasswordEncoder` 生成密码哈希。
2. 登录时使用 `encoder.matches(raw, encoded)` 校验。

**经验沉淀**：密码必须在早期就改为 BCrypt 存储，不要等数据积累后再迁移，否则会产生历史数据包袱。

### 7.3 跨服务调用权限死锁

**现象**：`Order-Service` 通过 Feign 调用 `Auth-Service` 获取用户信息时报 `403 Forbidden`，或服务间调用出现 401/403。

**原因**：Feign 内部调用默认不携带前端 Token；但 Auth/Flight 等服务默认开启 Security 拦截，导致内部调用被当成未授权请求。

**解决**：

1. 在 Auth 服务中定义 `/internal/**` 内部接口。
2. 在 `SecurityConfig` 中将 `/api/auth/internal/**` 放在高优先级规则中 `permitAll()`。
3. 项目整体保留 Gateway 做统一鉴权，下游服务信任 Gateway 传递的 Header。

**排查建议**：如果服务间调用被拒，先分清是“前端请求没 Token”还是“内部 Feign 被下游 Security 拦截”。

## 8. Feign 联调与服务契约问题

### 8.1 Feign 404

**现象**：Order 调 Flight 或 Auth 时返回 404。

**原因**：记录中出现过因数据库没有对应航班数据导致 Feign 查询失败。

**解决**：插入真实、可被接口查到的测试数据，再重新联调。

**排查建议**：Feign 404 不一定是路径错，也可能是路径正确但业务资源不存在。要同时检查 Controller 映射、Feign 路径和数据库测试数据。

### 8.2 Feign 反序列化失败

**现象**：Feign 调用时报 `DecodeException`。

**原因**：Flight Controller 返回 `List`，但 Feign Client 定义返回单个对象或 `Object`，接口契约不一致。

**解决**：修改 Controller，增加返回单个对象的方法，并让 Feign Client 的返回类型与实际响应结构一致。

**经验沉淀**：Feign 调用前先确认三件事：URL 一致、HTTP 方法一致、返回 DTO 一致。

## 9. 数据库、JPA 与领域模型问题

### 9.1 Jackson 双向关联死循环

**现象**：访问航班列表时后端 `StackOverflowError`，或前端迟迟收不到响应。

**原因**：`Flight` 实体引用 `Airline`，`Airline` 又反向引用 `List<Flight>`，Jackson 转 JSON 时陷入 A -> B -> A 的无限递归。

**解决**：在关联字段上添加 `@JsonIgnoreProperties` 或 `@JsonIgnore`，切断双向序列化链路。

**排查建议**：实体存在双向关联时，不要直接把 Entity 暴露给前端，优先使用 VO/DTO。

### 9.2 数据库保留字冲突

**现象**：保存订单时报 `500 Internal Server Error`，日志显示 SQL 语法错误。

**原因**：`tickets` 表中字段名 `class` 与 MySQL/Java 保留字冲突。

**解决**：在实体类映射中对字段名做转义。当时采用的是在 `@Column` 映射中用反引号包裹数据库字段名。

**后续建议**：从长期维护角度，优先把字段改成 `cabin_class` 之类的非保留字，减少 ORM 转义成本。

### 9.3 懒加载异常

**现象**：管理员能看航班，用户查航班时报 500。

**原因**：管理员接口直接查实体；用户接口需要把实体转成 VO，转换过程中访问了懒加载的 `airline` 对象，此时 Hibernate Session 已关闭。

**解决**：

1. Service 层添加 `@Transactional(readOnly = true)`。
2. Repository 查询中使用 `JOIN FETCH` 显式立即加载关联对象。

**排查建议**：如果“同一数据管理员能查、用户不能查”，要比较两条链路是否一个直接返回 Entity、另一个做了 VO 转换。

### 9.4 身份证加密后长度溢出

**现象**：注册用户时崩溃。

**原因**：数据库 `id_card` 原本是 `varchar(18)`；启用 AES 加密后，密文会变成 64 位以上的 Base64 字符串，写入时被截断或报错。

**解决**：将数据库字段和实体类中的 `idCard` 长度扩容到 `255`。

**经验沉淀**：敏感字段加密后，数据库长度要按密文长度设计，而不是按明文长度设计。

### 9.5 退票触发 Check 约束

**现象**：退票时报 `Check constraint 'chk_valid_payment_time' is violated`。

**原因**：数据库约束要求 `status='已取消'` 时 `payment_time` 必须为 `NULL`，但代码只修改了状态，没有清空支付时间。

**解决**：在 `refundTicket` 逻辑中显式执行 `ticket.setPaymentTime(null)`。

**排查建议**：遇到数据库约束错误时，不要只看 Java 代码，也要回看表结构中的 Check、Not Null、外键等约束。

## 10. Redis 缓存问题

### 10.1 `@Cacheable` 生效但 Redis 中没有 Key

**现象**：接口返回成功，但 Redis 执行 `keys *` 为空。

**原因**：Spring Boot 没有强制指定缓存类型，实际使用了本地内存缓存而非 Redis。

**解决**：

1. 在 Nacos 配置中添加：

```yaml
spring:
  cache:
    type: redis
```

2. 在启动类添加 `@EnableCaching`。

### 10.2 LocalDate 序列化导致缓存接口 500

**现象**：强制使用 Redis 后，接口报 500。

**原因**：`FlightSearchVO` 包含 Java 8 的 `LocalDate` 类型，Redis 默认序列化或普通 JSON 序列化无法正确处理。

**解决**：编写 `RedisConfig`，自定义 `ObjectMapper`，注册 `JavaTimeModule`，并禁用“写为时间戳”。

**经验沉淀**：缓存 DTO 中包含 `LocalDate`、`LocalDateTime` 时，要主动配置 Jackson Java Time 支持。

## 11. 订单与业务数据一致性问题

### 11.1 乘机人姓名匹配失败

**现象**：用户帮家人买票，订单中所有乘机人姓名都变成了订票人自己。

**原因**：

1. 前端传来的 ID 是 String，后端 DTO 是 Integer，数据库是 Long，类型不一致导致 `equals` 判断失败。
2. 前端传的是虚拟 ID，例如 `f_1`，后端拿这个 ID 去数据库查不到真实 `member_id`。

**解决**：放弃复杂的后端二次 ID 匹配，改为前端负责解析 ID 为姓名，直接把 `passengerNames` 列表发送给后端；后端只负责落库。

**经验沉淀**：跨端 ID 设计要统一类型和语义。虚拟 ID 适合前端交互，但不适合直接作为后端查询主键。

### 11.2 “一单多票”与服务解耦

**问题背景**：机票预订支持一次为多人购票。如果订单服务直接关联 User/Flight 实体，会造成跨服务耦合和后续查询复杂。

**解决思路**：订单服务保存乘机人快照、航班号、日期、舱位、价格等信息，不直接持有 User/Flight 实体关联。一次预订生成多张独立 `Ticket` 记录，再通过 `booking_time` 等逻辑聚合为一笔订单。

**后续注意**：阶段文档中提到 `daily_flight_stock` 可用于高并发库存扣减，当前设计书也把它列为后续演进方向；如果继续扩展并发下单，需要补库存扣减和分布式事务/补偿机制。

## 12. 前端工程化与 UI 问题

### 12.1 ECharts 依赖找不到

**现象**：集成 ECharts 时出现 `Module not found: Error: Can't resolve 'echarts'`。

**原因**：多模块项目中容易在根目录执行 npm 命令，但前端实际运行目录是 `frontend` 子目录，导致依赖没有安装到正确的 `node_modules`。

**解决**：

1. 定位报错路径，确认实际使用的 `node_modules`。
2. 进入 `frontend` 目录执行：

```bash
npm install echarts --save
```

3. 重启 Webpack Dev Server。
4. Vue 2 环境中使用 `import * as echarts from 'echarts'` 确保图表实例稳定。

### 12.2 ESLint 标签与空 catch 报错

**现象**：频繁出现 `x-invalid-end-tag` 和 `no-empty`。

**原因**：

1. Vue 模板中 `el-col`、`section`、`div` 等嵌套复杂，一个多余闭合标签就会导致编译失败。
2. 异步 API 的 `catch (e) {}` 空块违反 ESLint 规则。

**解决**：

1. 给所有 `catch` 块补充 `console.error` 或错误日志。
2. 重构 HTML 结构，将复杂嵌套扁平化。
3. 必要时用 CSS Grid 替代多层 Flexbox 嵌套。

### 12.3 Element UI 样式难覆盖

**现象**：`el-collapse-item__header`、`el-card__body` 等 Element UI 内部样式难以覆盖。

**原因**：Vue 单文件组件使用 `<style scoped>` 后，普通选择器无法触达子组件内部生成的 DOM。

**解决**：使用 `::v-deep` 深度选择器覆盖 Element UI 内部样式；对特殊卡片手动覆盖 `el-card__body` 内边距。

### 12.4 响应式布局坍塌

**现象**：在 1366px 等常见笔记本分辨率下，原本并排的卡片被挤压，文字溢出。

**原因**：管理端数据密度高，表格列数固定，早期布局没有充分考虑中小屏。

**解决**：

1. 引入媒体查询。
2. 小屏下将“常用乘机人”表格设为 `overflow-x: auto`。
3. 将 4 列指标卡片在手机端自动转为 2 列或 1 列。

## 13. 推荐排查流程

### 13.1 服务启动失败

1. 先看是否是 Maven 构建问题：插件、包名、Common 是否 install。
2. 再看 Nacos 是否连通：9848、standalone、实例 IP。
3. 再看配置是否齐全：数据库、Redis、JWT。
4. 最后看是否引入了不该引入的自动配置：Gateway 是否触发 DataSource。

### 13.2 Gateway 转发失败

1. 看 Nacos 服务实例是否存在。
2. 看实例 IP 是否是可访问的 `127.0.0.1` 或本机真实 IP。
3. 看 Gateway 路由是否指向 `lb://xxx-service`。
4. 看 Gateway 是否混入 Servlet Filter 或 Servlet 异常处理器。
5. 看 CORS 预检请求是否被拦截。

### 13.3 Feign 调用失败

1. 401/403：优先检查下游 Security 是否放行内部接口。
2. 404：同时检查路径和测试数据。
3. `DecodeException`：检查 Controller 返回结构与 Feign Client 返回类型。
4. DTO 字段异常：检查 Common 是否重新 install，服务是否重启。

### 13.4 业务接口 500

1. 如果是数据库写入，检查字段名、长度、约束。
2. 如果是查询转 JSON，检查双向关联和懒加载。
3. 如果是缓存接口，检查 Redis 序列化。
4. 如果是登录或鉴权，检查 JWT 配置和 BCrypt 密码来源。

## 14. 可复用开发规范

1. 修改 `common-module` 后必须执行 `mvn clean install -DskipTests`。
2. Gateway 只做路由、鉴权和跨域，不直接连数据库。
3. Gateway 使用 WebFlux 组件，不使用 Servlet Filter。
4. JWT 密钥配置要在所有使用 `JwtTokenProvider` 的服务中补齐。
5. 服务间 Feign 调用要有明确的内部接口和放行策略。
6. Feign Client、Controller、DTO 三者必须同步修改。
7. Entity 不建议直接返回前端，复杂关联优先转 VO/DTO。
8. 加密字段按密文长度设计数据库字段。
9. Redis 缓存对象包含时间类型时必须配置 Java Time 序列化。
10. 前端依赖安装要进入实际运行的 `frontend` 目录。
